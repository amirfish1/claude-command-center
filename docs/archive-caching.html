CCC archive cache architecture

How the all-repos archive, dashboard, and attention feeds stay fast: five cache layers sit between the pollers and the expensive O(all-sessions) archive builder.
Documents server.py as of 2026-07-17 (commit aedd0e10). Helper names are explanatory labels only, not a stable API. Every claim below was checked against the code, not the design notes.

The problem this stack solves: /api/sessions?all=1 and /api/conversations?all=1 are polled continuously by the dashboard and the COO board. A naive implementation re-walks and re-parses every transcript on every poll, and on a machine with years of agent history that is a multi-megabyte payload rebuilt in a 1-2 second GIL-bound burn, per request, uncoalesced. Each layer below removes one class of that repeated work while keeping a strict bound on how stale the served data can be.

Request flow

Scroll sideways on a narrow screen; the diagram keeps its desktop layout.

Read the diagram left to right. A polling request never travels further right than Tier 1: it gets a copy of the current snapshot immediately, and at most one background thread per cache key walks Tiers 2 and 3 to bring the snapshot up to date. Tier 3 runs only when the corpus signature says something a build actually reads has changed on disk.

The two-second serve window

Freshness bound for warm serves: transcript-derived fields (state, ended_blocked, waiting question) change as soon as a refresh sees the new signature; process-liveness and sidecar fields lag by at most the serve TTL plus one refresh, and the engine process scan adds up to 4s on top.

Tunable settings

CCC_ARCHIVE_SERVE_TTL_SEC env
default 2, minimum 0; 0 disables the serve cache entirely

Width of the stale-while-revalidate window in Tier 1, and also the refresh window of the spawned-session coalescing cache. Lowering it makes liveness chips fresher but multiplies rehydrate work under parallel polling; setting it to 0 makes every request compute rows synchronously, which restores the exact CPU burn this cache was built to kill. Raising it saves CPU at the cost of liveness lag; transcript-content freshness is unaffected because the signature still gates rebuilds.

CCC_ARCHIVE_SIG_TTL_SEC env
default 2, minimum 0; 0 disables the signature memo

How long one corpus fingerprint is shared by all cache keys. The fingerprint is a stat-only walk of every transcript, so with N keys refreshing concurrently an unmemoized signature means N full-corpus scandir passes per window; that sustained burn is what this memo removed. Keep it at or below the serve TTL: the serve path already tolerates that much staleness, so the memo adds no new correctness window.

CCC_ARCHIVE_CACHE_TTL_SEC env currently inert on the serve path
default 300, floor 30 (values below 30 are clamped up)

Freshness label for the persisted response cache: an entry older than this is considered stale by _archive_response_cache_is_stale.

Reality check (verified 2026-07-17): the helpers that consume this TTL (_archive_cached_payload and _archive_refresh_response_cache_async) have no callers in server.py. On the live serve path the persisted cache is invalidated purely by corpus-signature mismatch, never by age. Changing this variable today has no observable effect; it is documented because the env var still parses at startup and the constant would gate any future re-adoption of the TTL-based stale-serve path.

CCC_SESSION_LIVE_TTL_SEC env
default 2, minimum 0; 0 disables the liveness memo

Not one of the three archive TTLs, but it shapes the same hot path: the per-session liveness memo used during row rehydration. Liveness already lags by the serve TTL (2s) and the engine scan TTL (4s), so the default introduces no staleness the callers did not already have.

Invalidation rules

  • Transcript change: any add, edit, or delete under ~/.claude/projects flips a file's (mtime_ns, size), the corpus signature changes, and the next background refresh rebuilds. Sibling engine stores (codex, cursor, antigravity, hermes) are covered at directory or db-file granularity. The rebuild reuses the per-file parse cache, so only touched transcripts re-parse.
  • Lifecycle mutation: POST handlers that rename, pin, archive, or otherwise mutate session state call _clear_archive_serve_cache(). It empties the serve cache and bumps a generation counter, so a build that started before the mutation cannot store its now-stale rows.
  • Deliberately excluded from the signature: fast-changing live state (sidecar markers, the spawned registry). Those mutate on every poll while a session is live; folding them in would bust the build cache constantly and pin the CPU. They are re-layered onto rows at rehydrate and serve time instead (_stamp_archive_state, _stamp_archive_goals, watchtower display names), so they stay fresh without gating the expensive build.

What caching does not remove

The first cold build still happens

A truly first-ever run (no persisted cache on disk) builds synchronously and can take 20-30s on a large corpus. After a restart the persisted payload is served from disk immediately, rehydrated for fresh live state, while the real rebuild runs in the background.

The force path always runs the full builder

/api/conversations/all without stale_ok=1 is the explicit archive load (the one with the progress overlay). It bypasses every tier and invokes _build_archive_conversations directly on the request thread. It persists its result for cold-start serving, but without a corpus signature, so the next signature-gated refresh treats that entry as changed and rebuilds once in the background.

CCC-415: per-file incremental cache (not merged, not production)

Design alternative only. Nothing on this page describes it.

Branch codex/ccc-415-incremental-archive-cache (single commit 1d86e49b, "perf(archive): add per-file archive cache", roughly 540 changed lines in server.py plus perf-budget test edits) explores a different shape: instead of one whole-response payload gated by one global corpus signature, it caches archive rows per transcript file and invalidates each file independently.

It is not merged and does not run in any CCC build. The production stack documented above landed first and removed the measured hot-path cost with a much smaller diff, so the per-file branch never had a remaining bottleneck to justify its extra surface. Merging it now would mean reconciling two overlapping invalidation models (per-file entries versus the corpus signature plus per-file parse cache that ships today) and re-baselining the call-count invariants in tests/test_perf_budget.py. Treat it as a design note for a future rework, not as behavior.
